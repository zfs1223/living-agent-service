# 人工干预决策机制设计

> 基于生命企业智能体进化成长核心概念的人工干预必要性分析

---

## 一、设计理念

### 1.1 核心原则

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    人工干预设计原则                                            │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  【生命智能体进化成长核心概念】                                                 │
│                                                                             │
│  1. 自主进化优先                                                             │
│     ├── AI能自主完成的，不干预                                               │
│     ├── AI能学习改进的，给机会学习                                            │
│     └── AI能力边界的，才需要人工介入                                          │
│                                                                             │
│  2. 风险导向决策                                                             │
│     ├── 低风险决策 → AI自主执行                                              │
│     ├── 中风险决策 → AI执行 + 事后通知                                       │
│     └── 高风险决策 → AI建议 + 人工确认                                       │
│                                                                             │
│  3. 成长型思维                                                               │
│     ├── 错误是学习机会，不是失败                                              │
│     ├── 人工干预是"教学"，不是"纠正"                                         │
│     └── 每次干预都应提升AI能力                                               │
│                                                                             │
│  4. 最小干预原则                                                             │
│     ├── 干预是例外，不是常态                                                 │
│     ├── 干预成本应小于自动化收益                                             │
│     └── 干预决策应有明确规则                                                 │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

```
│  │   ├─────────────────┼─────────────────┼─────────────────┤               │

│  │   └─────────────────┴─────────────────┴─────────────────┘               │
│  │                                                                          │
│  └───────────────────────────────────────────────────────────────────────→  │
│                          风险等级                                            │
│         低 ←─────────────────────────────────────────────→ 高               │
│                                                                             │
│  【决策规则】                                                                │
│  ├── 风险高 + 影响大 → 必须人工确认                                          │
│  ├── 风险中 + 影响大 → 建议人工确认                                          │
│  ├── 风险低 + 影响大 → AI执行 + 通知                                         │
│  ├── 风险高 + 影响小 → AI执行 + 通知                                         │
│  └── 风险低 + 影响小 → AI自主执行                                            │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

```java
@Service
public class InterventionDecisionEngine {
    
    private final InterventionRuleRegistry ruleRegistry;
    private final RiskAssessmentService riskService;
    private final ImpactAnalyzer impactAnalyzer;
    
    public InterventionDecision decide(InterventionContext context) {
        // Step 1: 风险评估
        RiskLevel risk = riskService.assess(context);
        
        // Step 2: 影响分析
        ImpactLevel impact = impactAnalyzer.analyze(context);
        
        // Step 3: 规则匹配
        List<InterventionRule> matchedRules = ruleRegistry.match(context);
        
        // Step 4: 决策生成
        if (hasMandatoryRule(matchedRules)) {
            return InterventionDecision.mandatory(
                getMandatoryRule(matchedRules),
                risk,
                impact
            );
        }
        
        // Step 5: 矩阵决策
        return decideByMatrix(risk, impact, context);
    }
    
    private InterventionDecision decideByMatrix(
            RiskLevel risk, 
            ImpactLevel impact,
            InterventionContext context) {
        
        // 决策矩阵
        if (risk == RiskLevel.HIGH && impact == ImpactLevel.HIGH) {
            return InterventionDecision.mandatory(
                "高风险高影响，必须人工确认",
                InterventionMode.REALTIME_CONFIRM
            );
        }
        
        if (risk == RiskLevel.HIGH || impact == ImpactLevel.HIGH) {
            return InterventionDecision.recommended(
                "高风险或高影响，建议人工确认",
                InterventionMode.ASYNC_APPROVAL
            );
        }
        
        if (risk == RiskLevel.MEDIUM && impact == ImpactLevel.MEDIUM) {
            return InterventionDecision.notify(
                "中等风险影响，AI执行后通知",
                InterventionMode.PERIODIC_REVIEW
            );
        }
        
        return InterventionDecision.autonomous(
            "低风险低影响，AI自主执行",
            InterventionMode.AUTO_EXECUTE
        );
    }
}
```

### 4.3 干预流程编排

```java
@Service
public class InterventionOrchestrator {
    
    private final NotificationService notificationService;
    private final InterventionDecisionEngine decisionEngine;
    private final InterventionHistoryRepository historyRepository;
    
    public InterventionResult execute(InterventionRequest request) {
        // Step 1: 决策
        InterventionDecision decision = decisionEngine.decide(
            InterventionContext.from(request)
        );
        
        // Step 2: 记录
        historyRepository.record(decision);
        
        // Step 3: 执行
        return switch (decision.getMode()) {
            case REALTIME_CONFIRM -> executeRealtimeConfirm(decision, request);
            case ASYNC_APPROVAL -> executeAsyncApproval(decision, request);
            case PERIODIC_REVIEW -> executePeriodicReview(decision, request);
            case AUTO_EXECUTE -> executeAutonomous(decision, request);
            case HUMAN_LEAD -> executeHumanLead(decision, request);
            case HUMAN_EXECUTE -> executeHumanExecute(decision, request);
        };
    }
    
    private InterventionResult executeRealtimeConfirm(
            InterventionDecision decision, 
            InterventionRequest request) {
        
        // 发送实时确认请求
        ConfirmRequest confirmRequest = ConfirmRequest.builder()
            .title(decision.getTitle())
            .description(decision.getDescription())
            .riskLevel(decision.getRisk())
            .impactLevel(decision.getImpact())
            .aiSuggestion(decision.getAiSuggestion())
            .timeout(decision.getTimeout())
            .build();
        
        // 多渠道通知
        notificationService.sendUrgent(
            decision.getAssignee(),
            confirmRequest,
            Channel.PHONE, Channel.SMS, Channel.IM
        );
        
        // 等待响应
        ConfirmResponse response = waitForResponse(
            confirmRequest.getId(),
            decision.getTimeout()
        );
        
        if (response == null) {
            // 超时处理
            return handleTimeout(decision, request);
        }
        
        return InterventionResult.from(response);
    }
    
    private InterventionResult executeAsyncApproval(
            InterventionDecision decision,
            InterventionRequest request) {
        
        // 创建审批任务
        ApprovalTask task = ApprovalTask.builder()
            .title(decision.getTitle())
            .description(decision.getDescription())
            .requester("AI:" + request.getNeuronId())
            .assignee(decision.getAssignee())
            .priority(decision.getPriority())
            .deadline(Instant.now().plus(decision.getTimeout()))
            .build();
        
        // 加入审批队列
        approvalQueue.add(task);
        
        // 发送通知
        notificationService.send(
            decision.getAssignee(),
            task,
            Channel.IM, Channel.EMAIL
        );
        
        return InterventionResult.pending(task);
    }
    
    private InterventionResult executeAutonomous(
            InterventionDecision decision,
            InterventionRequest request) {
        
        // AI自主执行
        ExecutionResult result = request.execute();
        
        // 记录日志
        autonomousLogRepository.record(
            AutonomousLog.builder()
                .request(request)
                .decision(decision)
                .result(result)
                .timestamp(Instant.now())
                .build()
        );
        
        // 根据配置决定是否通知
        if (decision.shouldNotify()) {
            notificationService.send(
                decision.getNotifyTarget(),
                AutonomousExecutionNotification.from(result),
                Channel.SYSTEM
            );
        }
        
        return InterventionResult.success(result);
    }
}
```

```sql
    priority INTEGER DEFAULT 50,
```

## 十二、与合规管理系统的集成

> 与 `21-compliance-optimization.md` 的集成设计

### 12.1 集成背景

合规管理系统需要与人工干预机制深度集成，确保：

1. **审计追溯**: 所有关键操作都有完整的审计日志
2. **合规预检**: 在人工干预前完成合规检查
3. **规则隔离**: 合规类规则不应被 AI 学习改变

### 12.4 不可学习规则机制

合规类规则必须标记为 `learningEnabled: false`，原因：

1. **法律确定性**: 合规规则来自法律法规，不应通过 AI 学习改变
2. **责任归属**: 合规违规责任由人工承担，AI 不能改变规则
3. **审计要求**: 审计需要明确的规则依据，不能是"学习结果"

```java
// 修改后的学习机制
@Service
public class InterventionLearningService {
    
    public void learnFromIntervention(InterventionDecision decision) {
        // 检查规则是否允许学习
        InterventionRule rule = ruleRepository.findById(decision.getRuleId());
        
        if (!rule.isLearningEnabled()) {
            // 合规类规则不学习，仅记录
            log.info("规则 {} 标记为不可学习，跳过学习流程", rule.getRuleId());
            return;
        }
        
        // 正常学习流程
        doLearn(decision);
    }
}
```

## 十三、总结

### 核心设计原则

1. **自主进化优先**: AI能做的，不干预；AI能学的，给机会学
2. **风险导向决策**: 根据风险和影响决定干预方式
3. **成长型思维**: 每次干预都是学习机会
4. **最小干预原则**: 干预是例外，不是常态

### 关键创新点

1. **干预决策矩阵**: 风险×影响 → 决策模式
2. **四类干预场景**: 必须干预、建议干预、AI自主、学习扩展
3. **四种交互模式**: 实时确认、异步审批、定期审核、异常上报
4. **学习进化闭环**: 干预 → 学习 → 能力提升 → 自主范围扩展
5. **合规集成机制**: 合规预检 + 不可学习规则 + 审计日志

### 优化建议处理结果

| 优化建议 | 必要性 | 处理方式 |
|---------|--------|---------|
| 人工保留场景 | ★★★ 核心 | 本文档已完整覆盖 |
| AI化程度评估 | ★★★ 核心 | 第十一章详细设计 |
| 业务流程编排 | ★★☆ 重要 | 第十一章详细设计(学习机制) |
| 岗位级KPI | ★☆☆ 次要 | 动态生成机制(可选实现) |
| 岗位体系扩展 | ★☆☆ 次要 | 自然涌现机制(不建议预设) |
| 实施路径对齐 | ★☆☆ 次要 | 自适应机制(不建议固定) |
| 合规管理集成 | ★★★ 核心 | 第十二章详细设计 |

### 与现有框架关系

- **不冲突**: 本设计是对现有框架的补充，不改变核心架构
- **不重复**: 本设计专注于"人工干预决策"，与现有模块职责清晰
- **可集成**: 通过明确的集成点与进化系统、主动预判、运营评判、合规管理无缝对接
